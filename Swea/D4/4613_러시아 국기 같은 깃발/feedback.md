보내주신 링크는 이전 햄버거 다이어트 문제의 링크지만, 설명해 주신 기발한(?) 접근법을 보니 SWEA 4613 '러시아 국기 같은 깃발' 문제군요! 정확하게 짚어오셨습니다.

먼저, 스스로 고안하신 그 접근법에 박수를 쳐드리고 싶습니다. "B가 가장 많은 곳을 기준으로 삼고 위아래로 확장한다"는 방식은 알고리즘에서 매우 중요한 기법 중 하나인 **'그리디(Greedy, 탐욕법)'** 의
전형적인 사고방식입니다. C++ 솔루션 개발 등 실무에서 빠르고 합리적인 최적해를 찾을 때 자주 쓰이는 훌륭한 감각이죠.

하지만 안타깝게도 이 문제에는 치명적인 반례가 존재합니다.
가장 B가 많은 줄을 B로 칠하고 넘어가려 했는데, 막상 그렇게 기준을 잡아버리면 위쪽의 W 구역이나 아래쪽의 R 구역에서 기존 색을 너무 많이 덮어씌워야 해서 전체 색칠 비용(페인트 값)이 오히려 폭증해버리는
상황이 발생할 수 있습니다. 국지적인 이득이 전체의 최적해를 보장하지 못하는 것이죠.

이럴 땐 어떻게 해야 할까요? SSAFY 코테에서 가장 사랑하는 방식, '그냥 무식하게 선을 다 그어보는(완전 탐색)' 접근으로 돌아가야 합니다. 약속된 5단계로 패턴을 완벽히 부숴드리겠습니다.

---

### 1. 🔍 요구사항 분석 및 유형 파악

* **문제 유형:** 완전 탐색 (Brute-Force), 구간 나누기
* **유형 파악 키워드:** "위에서 몇 줄", "그 다음 몇 줄", "나머지 줄", "N과 M은 50 이하이다."
    * **핵심 단서:** 데이터의 크기(N, M)가 최대 50으로 매우 작습니다. 구역을 나누는 모든 경우의 수를 다 해봐도 시간이 널널하다는 강력한 힌트입니다.
* **핵심 목표:** W, B, R 세 구역으로 나누는 **'경계선 2개'** 의 위치를 정하여, 새로 칠해야 하는 칸의 개수를 최소화하기.
* **입력 범위 및 시간 복잡도:** 경계선 2개를 긋는 경우의 수는 약 1,200가지입니다. 각 경우마다 50x50 격자를 확인해도 약 300만 번의 연산이므로 1초(약 2천만 번) 안에 여유롭게 통과합니다.

### 2. 🧠 논리적 설계 (주석 뼈대)

가장 많은 것을 찾는 그리디 방식은 버리고, **흰색(W)이 끝나는 줄**과 **파란색(B)이 끝나는 줄**을 결정하는 2중 `for`문 뼈대를 세웁니다.

```markdown
1. 테스트 케이스 T를 입력받는다.
2. N(행), M(열)을 입력받고, 국기 모양 문자열을 리스트로 저장한다.
3. 최소 색칠 횟수를 저장할 변수(min_paint)를 무한대 값으로 초기화한다.
4. 흰색(W) 구역이 끝나는 행(i)을 0부터 N-3까지 순회한다. (최소 1줄씩은 가져야 하므로)
5. 파란색(B) 구역이 끝나는 행(j)을 i+1부터 N-2까지 순회한다.
   (이때 자동으로 빨간색(R) 구역은 j+1부터 N-1까지가 된다.)
6. 정해진 i, j 경계선에 따라 각 구역을 돌면서,
    - 0 ~ i행: W가 아닌 칸의 개수 카운트
    - i+1 ~ j행: B가 아닌 칸의 개수 카운트
    - j+1 ~ N-1행: R이 아닌 칸의 개수 카운트
7. 카운트된 총합이 min_paint보다 작으면 갱신한다.
```

### 3. 💻 코드 구현 (Pythonic)

이해를 돕기 위해 뼈대를 그대로 코드로 옮긴 가장 직관적인 형태입니다.

```python
T = int(input())

for t in range(1, T + 1):
    N, M = map(int, input().split())
    flag = [input().strip() for _ in range(N)]

    # 파이썬에서 최솟값을 갱신할 때 자주 쓰는 초기값 설정
    min_paint = float('inf')

    # i: W(흰색) 구역이 끝나는 행 (0 ~ N-3)
    for i in range(N - 2):
        # j: B(파란색) 구역이 끝나는 행 (i+1 ~ N-2)
        for j in range(i + 1, N - 1):

            paint_cnt = 0

            # 1. W 구역 (0행 ~ i행) 검사: 'W'가 아닌 것의 개수를 더함
            for w_row in range(0, i + 1):
                paint_cnt += M - flag[w_row].count('W')

            # 2. B 구역 (i+1행 ~ j행) 검사: 'B'가 아닌 것의 개수를 더함
            for b_row in range(i + 1, j + 1):
                paint_cnt += M - flag[b_row].count('B')

            # 3. R 구역 (j+1행 ~ N-1행) 검사: 'R'이 아닌 것의 개수를 더함
            for r_row in range(j + 1, N):
                paint_cnt += M - flag[r_row].count('R')

            # 최솟값 갱신
            if paint_cnt < min_paint:
                min_paint = paint_cnt

    print(f"#{t} {min_paint}")
```

### 4. 🛡️ 엣지 케이스 점검

* **"각 색깔은 최소 1줄 이상이어야 한다."**
  이 조건이 코딩테스트에서 가장 많이 실수하는 포인트입니다. `for`문의 범위를 `i`는 `N-2` 전까지, `j`는 `N-1` 전까지로 제한함으로써 W, B, R이 무조건 최소 1줄씩은 공간을 차지하도록
  완벽하게 강제했습니다.

### 5. 🏆 최고 수준 최적화 (Review & Optimization)

위의 코드는 정답이지만, 매번 범위를 나눌 때마다 `.count()` 함수가 각 줄을 처음부터 다시 읽으면서 개수를 셉니다. 중복 연산이 꽤 많이 발생하죠.
**"각 줄을 W, B, R로 칠할 때 드는 비용을 미리 다 계산해 둔다면?"** 이라는 발상의 전환을 적용하면 압도적으로 빠른 코드가 탄생합니다.

```python
T = int(input())

for t in range(1, T + 1):
    N, M = map(int, input().split())
    flag = [input().strip() for _ in range(N)]

    # [최적화 1] 각 줄을 W, B, R로 바꿀 때 칠해야 하는 개수(비용)를 미리 계산
    cost_W = [M - row.count('W') for row in flag]
    cost_B = [M - row.count('B') for row in flag]
    cost_R = [M - row.count('R') for row in flag]

    min_paint = float('inf')

    for i in range(N - 2):
        for j in range(i + 1, N - 1):
            # [최적화 2] 매번 count()를 돌지 않고, 미리 구해둔 리스트의 부분합만 구함
            current_paint = sum(cost_W[:i + 1]) + sum(cost_B[i + 1:j + 1]) + sum(cost_R[j + 1:])

            if current_paint < min_paint:
                min_paint = current_paint

    print(f"#{t} {min_paint}")
```

**💡 최적화 핵심 원리:**
이 기법을 **'전처리(Preprocessing)'** 또는 '누적 비용 배열'이라고 합니다.
경계선을 1,000번 그어보더라도, "3번째 줄을 파란색으로 칠하는 비용"은 언제나 똑같습니다. 변하지 않는 값을 미리 계산해 두고, 나중에는 파이썬의 강력한 내장 함수 `sum()`과 슬라이싱(`[:i+1]`)으로
꺼내 쓰기만 하면 실행 시간이 극적으로 줄어듭니다.
